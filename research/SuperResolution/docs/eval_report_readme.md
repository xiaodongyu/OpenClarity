# How to Read the HTML Eval Report

This explains how to interpret `test/fixtures/medicine_package/eval_<date>_<seq>.html`,
generated by `test/eval_medicine_package.py`.

## Scale factors (×2, ×3, ×4) and the ground-truth downscale

x2/x3/x4 refer to the **magnification (scale) factor** — how many times
larger the output image is than the low-resolution input, in each dimension.
×4 is also the largest scale with a public pretrained FSRCNN/ESPCN
checkpoint (see `docs/algorithm_readme.md` §2.5) — there's no ×8+ pretrained
model to compare against yet.

The eval pipeline, for each fixture image:

1. Takes the real photo, centre-crops it to `CROP_SIZE` (800×600), then
   shrinks that crop by `BASE_DOWNSCALE` (×4) to get the actual "ground
   truth" reference used for comparison (200×150). This extra downscale step
   exists because at 800×600 the crop is still sharp enough that even ×2-×4
   SR barely looks different from the original — shrinking the reference
   first makes the same nominal scale factors correspond to a harder, more
   visibly-different reconstruction problem.
2. For each scale, shrinks that 200×150 ground truth further to make a
   synthetic low-res input — e.g. at ×4, it becomes 50×37.
3. Has each method (bilinear, SR model, etc.) blow that low-res image back up
   to 200×150.
4. Compares the reconstruction against the 200×150 ground truth to get
   PSNR/SSIM.

So, relative to the 200×150 ground truth:
- **×2** = recover it from a 100×75 input (mild downsampling, easiest to
  reconstruct well — hence the highest PSNR numbers).
- **×3** = recover it from a ~67×50 input (more information lost).
- **×4** = recover it from a 50×37 input (most information lost, hardest —
  hence the lowest PSNR numbers).

This maps directly to the real use case: if a low-vision user "zooms 4×"
into a region, the camera/crop only has 1/4 the linear resolution of what
they're viewing on screen, and the SR model's job is to fill in that missing
detail plausibly.

**Scales aren't comparable to each other numerically** — a ×2 PSNR of 28 dB
and a ×4 PSNR of 23 dB aren't "×2 is 5 dB better," they're different, harder
problems, since ×4 throws away far more information before reconstruction.

ESPCN's public pretrained checkpoint only exists at ×3 upstream (not ×2/×4),
so it appears alongside `fsrcnn_x3` in the ×3 group, while ×2 and ×4 only
have an FSRCNN entry.

## What "ground truth" actually is (and isn't)

PSNR/SSIM are computed against the **downscaled reference** described above
(the 200×150 image, after the `CROP_SIZE` crop and `BASE_DOWNSCALE` shrink) —
**not** against the 800×600 crop, and not against the original ~4032×3024
photo. Concretely, `test/eval_medicine_package.py` only ever writes the
already-shrunk 200×150 image to the temp directory that `src/benchmark.py`'s
`evaluate()` reads; the original photo and the 800×600 crop are discarded
after that point and never touched again.

This matters because the 200×150 reference is itself already a lossy,
blurred version of reality — whatever fine detail existed in the original
photo but got thrown away by the ×4 `BASE_DOWNSCALE` step is **not being
tested for recovery at all**. The PSNR/SSIM numbers measure how well each
method reconstructs *that intermediate reference*, not fidelity to the true
original scene. Two consequences:

- Numbers from different `BASE_DOWNSCALE` settings aren't comparable to each
  other — e.g. the `BASE_DOWNSCALE=2` runs (`eval_20260701_002`/`_003`/`_004`,
  400×300 reference) and the `BASE_DOWNSCALE=4` runs (`eval_20260701_005`
  onward, 200×150 reference) use a different ground truth, so a ×4 PSNR from
  one isn't directly comparable to a ×4 PSNR from the other, even though both
  are labeled "×4."
- A higher `BASE_DOWNSCALE` makes the SR-vs-bilinear gap more visible (per
  the earlier request that led to this change) precisely because it's
  measuring reconstruction against a reference that's already missing detail
  the original photo had — it's a deliberate tradeoff between "visible
  difference in the report" and "how close to the true original scene," not
  a free improvement in measurement fidelity.

## Report layout

**Header (dark bar at top)**
- Title + the git commit subject/hash the run was tested against, plus the
  run ID (`eval_<date>_<seq>`) and timestamp — lets you tell which code
  version produced this report if you're comparing across runs.

**Summary banner (green/orange box)**
- Big `N / M` count = how many **image×scale combinations** had their best
  method beat bilinear PSNR.
- Green "PASS" = every combination improved; orange "NEEDS IMPROVEMENT" = at
  least one didn't. This is a coarse pass/fail gate, not a quality score —
  read the actual PSNR numbers below for magnitude.

  (If a scale has no pretrained model at all — e.g. a future ×8 run — its
  cards get a gray "Baseline only" badge instead and are excluded from this
  count, since there's no SR method to compare against bilinear yet.)

**One card per image×scale combination**
- **Card header**: filename + scale (e.g. `IMG_4255.JPG — x2`), and an
  "Improved"/"No improvement" badge — the per-card version of the summary
  verdict above.
- **Filmstrip row**: thumbnails left to right —
  - `ground truth`: the real reference crop, captioned with its resolution
    (e.g. `200×150`).
  - `low-res input`: the actual low-res image the models had to reconstruct
    from, upscaled with nearest-neighbor so its blockiness stays visible
    rather than being smoothed away. Captioned with its *native* resolution
    (e.g. `50×37 native`) and its own PSNR/SSIM against ground truth —
    this quantifies the crudest possible "before" baseline (nearest-neighbor,
    worse than bilinear), so you have a numeric anchor for how much every
    method actually recovers.
  - every method tested at that scale, each labeled with its name and
    `PSNR dB / SSIM` underneath. The method whose name is colored **green**
    is the best-scoring one for that card; **gray** labels are the
    `bilinear`/`adaptive_sharpen` baselines. Visually, look for which
    thumbnail actually looks sharper/more legible to you — PSNR/SSIM are
    proxies, not a substitute for eyeballing whether small text is readable.
- **Metrics table below the filmstrip**: same numbers as the filmstrip
  captions, but as a table (sorted by PSNR descending) with latency added —
  useful for the actual dB deltas rather than reading them off tiny
  thumbnails.

## Reading the metrics

- **PSNR (dB)**: higher = closer to the real ground truth pixel-for-pixel.
  A ~2 dB gap is a modest but real difference.
- **SSIM**: 0–1, higher = more structurally similar (edges/texture), less
  sensitive to small pixel-level shifts than PSNR — useful as a second
  opinion when PSNR and visual quality disagree.
- **Latency (ms)**: per-image processing time for that method on the machine
  the eval ran on — `bilinear` is near-instant (<1ms), SR models are
  tens-to-hundreds of ms.
- **`_hybrid` entries**: as of the first baseline run (`eval_20260701_001`),
  these score much *lower* than the plain SR model on every card — an
  artifact of `enhance()`'s fixed sharpening strength over-sharpening an
  already-good SR output, not a meaningful comparison yet (see
  `docs/algorithm_readme.md` §6 and GitHub issue #30). Don't read a low
  hybrid score as "hybrid mode is broken for all inputs" — it's specifically
  miscalibrated for *already-sharp* input.

## What to do with a report

Use it as a before/after gate when changing the SR model (new architecture,
fine-tuned weights, quantization) — re-run `test/eval_medicine_package.py`,
and check whether the `N/M` improved-count holds and whether per-method PSNR
moved in the expected direction. A new run writes a new
`eval_<date>_<seq>.html`/`.json` pair rather than overwriting the previous
one, so past reports stay around for comparison.
